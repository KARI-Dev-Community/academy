# Kari Dev Contribution Guide

Thanks for contributing to Kari Dev! This guide explains our workflows and standards.

## Workflow

1. Fork the repository
2. Create a descriptive feature branch (`git checkout -b feature/my-change`)
3. Make your changes following our style guides
4. Commit with clear messages
5. Push and open a Pull Request
6. A mentor will review your work
7. Address feedback and get merged

## Style Guides

- Markdown: use sentence case for headings
- Code: follow the existing codebase conventions
- Links: use relative paths for internal links

## Pull Request Template

```markdown
## Description

Brief description of changes.

## Type of change

- [ ] Documentation
- [ ] Bug fix
- [ ] New feature
- [ ] Refactor

## Checklist

- [ ] My code follows the style guidelines
- [ ] I have updated the documentation
- [ ] I have added tests that prove my fix/feature works
```

## Labels

| Label | Description |
|-------|-------------|
| `good-first-project` | Great for first-time contributors |
| `beginner` | Beginner-friendly task |
| `help-needed` | Needs community input |
| `mentor-review` | Waiting for mentor review |
| `documentation` | Documentation change |
| `bug` | Something is broken |
| `feature-request` | New feature suggestion |
