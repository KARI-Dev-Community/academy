# Mentor Guide

Welcome to the Kari Dev Mentor Program. This guide outlines responsibilities, standards, and processes for mentors.

## Mentor Responsibilities

- Review pull requests from learners in a timely manner
- Provide constructive, actionable feedback
- Maintain code quality standards
- Guide learners through blockers
- Update documentation as needed
- Participate in community discussions

## Review Checklist

Before approving a PR, verify:

- [ ] Code follows the project's style guide
- [ ] All tests pass
- [ ] Documentation is updated
- [ ] No hardcoded secrets or credentials
- [ ] Commit messages are clear and conventional
- [ ] The solution teaches the intended concepts
- [ ] Changes are minimal and focused

## Code Review Standards

1. **Be constructive**: Focus on improvement, not criticism
2. **Explain why**: Help learners understand the reasoning
3. **Keep it focused**: Review the scope of the PR, not unrelated code
4. **Use labels**: Apply `mentor-review` when waiting for mentor feedback
5. **Mark as approved**: Use "[approved]" or "[LGTM]" in comments when satisfied

## Becoming a Mentor

To become a mentor you must:

- Complete a minimum of 5 projects
- Have at least 10 merged pull requests
- Demonstrate good communication in issues and discussions
- Be approved by existing maintainers

Apply by opening an issue with the `mentor-application` label.
